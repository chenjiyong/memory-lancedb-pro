# 二、真实使用场景优化：以“工作连续性”替代“单条记忆连续性”

你第 2 条实际上把优化目标说得很清楚了：

> 我在当前 session 做开发，切到新 session，也能继续使用记忆记录开发进度和偏好（skills，Claude 工具，资源）

这和我上一版方案相比，重点发生了变化：

- 上一版更偏“session commit / archive”
- 这次更应偏“**working set continuity**”

## 2.1 三类用户模型

### A. 开发用户
需要记住的不是只有偏好，还包括：
- 当前项目目标
- 最近在改什么
- 已确认的实现决策
- 未完成 open loops
- 常用 skills / Claude tools
- 当前重要资源与文件线索

### B. 学习用户
需要记住：
- 正在学什么
- 已掌握 / 未掌握
- 偏好的解释风格
- 最近资料和例子
- 下一次继续时的学习上下文

### C. 搜资 / 调研用户
需要记住：
- 研究主题
- 已收集来源
- 来源可靠性/偏好
- 已抽取结论
- 尚待验证的问题

## 2.2 这意味着当前项目缺的不是“更多普通 memory”
缺的是 **resume packet / working set memory**。

## 2.3 修订后的重点设计
不建议新增一整套 OpenViking 式文件系统层，而建议在现有结构上加一个新的“工作连续性层”：

### 新增一类二级语义维度
不改现有 6 类主分类兼容性，新增 metadata facet：
- `activity_domain`: `dev | learning | research`
- `artifact_kind`: `tool | skill | resource | progress | preference | open_loop`
- `resume_priority`
- `resource_refs`
- `tool_refs`
- `skill_refs`
- `project_key`

这样可以保留现有 schema 主体，只增强检索语义。

# 三、安装 / 更新 / 暂停后恢复：要做“状态接管”，不是只做“安装成功”

这是这次方案里最该补强的地方。

因为你第 3 条不是“安装脚本更强”，而是：

> 插件在 fresh install、upgrade、idle 后恢复时，都能正常接管当前环境的记忆

## 3.1 当前已有基础
已有可复用基础：
- `validateStoragePath()` 与 startup checks
- `memory-upgrader`
- `migrate`
- `session-recovery`
- `openclaw-integration-playbook`
- setup repo 的：
  - schema filter
  - workspace fallback
  - plugin path detection
  - selfcheck / validate
  - upgrade / uninstall

## 3.2 修订后的方向
不要把安装脚本逻辑原样塞进插件，而是拆成两层：

### 层 A：安装/升级脚本负责“外部修复”
参考：
- `/Users/jige/work/toolbox/memory-lancedb-pro-setup/setup-memory.sh`

负责：
- `plugins.load.paths`
- `plugins.allow`
- slot 绑定
- schema 过滤
- gateway restart
- selfcheck

### 层 B：插件运行时负责“内部接管”
插件自身在启动 / bootstrap / 首次 turn 时应做：
- 当前 OpenClaw 版本检查
- 当前 workspace / agent workspace map 发现
- 当前 memory store 状态检查
- 是否需要 `upgrade` / `migrate` / `reembed`
- 最近 reflection/session artifact 扫描
- 空库但已有 workspace artifacts 时的 rehydrate 建议/行为

## 3.3 需要新增的能力
建议新增一个统一的 runtime rehydration 流程：

### `rehydration`
触发时机：
- plugin start
- `agent:bootstrap`
- 首次真实 turn
- 长时间 idle 后恢复

职责：
1. 检查环境
2. 检查数据平面
3. 识别“我是 fresh install / upgrade / resume / stale state”
4. 生成恢复策略

这里我会修正上一版方案的一个点：

**不要默认把 workspace 现有 markdown / reflection 全量回灌进 LanceDB。**
